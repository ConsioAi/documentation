Every request to the Consio API must be authenticated using an API key. Consio uses Bearer token authentication — you generate a key in your dashboard and include it in the `Authorization` header of each HTTP request. No session cookies or OAuth flows are required.

## Getting Your API Key

1. Log in to your [Consio dashboard](https://app.consio.dev).
2. Navigate to **Settings → API Keys**.
3. Click **Generate key**.
4. Copy the key — it begins with the prefix `csk_`.

<Warning>
  Your API key is shown **only once** immediately after generation. Copy it and store it in a secure location such as a password manager or secrets vault before closing the dialog. If you lose the key, you must revoke it and generate a new one.
</Warning>

## Passing the API Key

Include your API key as a Bearer token in the `Authorization` header of every request:

```http
Authorization: Bearer csk_your_api_key
```

### Code Examples

<CodeGroup>

```bash curl
curl --request GET \
  --url 'https://api-martin.consio.dev/public/v1/your_workspace_slug/calls' \
  --header 'Authorization: Bearer csk_your_api_key'
```

```python Python
import requests

url = "https://api-martin.consio.dev/public/v1/your_workspace_slug/calls"
headers = {"Authorization": "Bearer csk_your_api_key"}

response = requests.get(url, headers=headers)
print(response.json())
```

```javascript JavaScript / Node.js
const response = await fetch(
  "https://api-martin.consio.dev/public/v1/your_workspace_slug/calls",
  {
    method: "GET",
    headers: {
      Authorization: "Bearer csk_your_api_key",
    },
  }
);

const data = await response.json();
console.log(data);
```

</CodeGroup>

## Workspace Scoping

Each API key belongs to exactly one workspace. You cannot use the same key to access data from a different workspace — the `workspace_slug` in the request path must match the workspace that issued the key. Attempting to use a key from workspace A to query workspace B returns HTTP `403 Forbidden`.

If you operate multiple workspaces, generate a separate API key for each one.

## Authentication Error Reference

| Status | When it occurs |
|--------|----------------|
| `401` | No `Authorization` header was provided, or the value does not follow the `Bearer csk_...` format |
| `403` | The key is structurally valid but does not belong to the workspace identified by `workspace_slug` |

<Tip>
  Store your API key in an environment variable rather than hardcoding it in your source code:

  ```bash
  export CONSIO_API_KEY=csk_your_api_key
  ```

  Then reference it in your code:

  ```python Python
  import os
  api_key = os.environ["CONSIO_API_KEY"]
  ```

  ```javascript JavaScript / Node.js
  const apiKey = process.env.CONSIO_API_KEY;
  ```

  This keeps credentials out of version control and makes it easy to rotate keys without touching your codebase.
</Tip>

## Security Best Practices

- **Rotate keys regularly.** Periodically revoke existing keys and generate fresh ones to limit the blast radius of a potential leak.
- **Use separate keys per integration.** If one integration is compromised, you can revoke only that key without disrupting others.
- **Revoke unused keys.** Remove keys for integrations that are no longer active. You can manage all keys in **Settings → API Keys** in your dashboard.
